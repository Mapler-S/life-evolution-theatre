可视化方案旨在解答的问题

地球 46 亿年的历史中，生命经历过至少 5 次大规模灭绝，每一次都重塑了演化走向。 这段"深时间"（deep time）尺度远超人类直觉：用文字描述二叠纪末 96% 的海洋物种灭绝，读者只能抽象地点头； 但如果能亲眼看到腕足动物的分支如何在 252 Ma 的那一刻断裂、三叶虫的化石点如何从繁盛的泛大陆海岸线骤然消失， 这种"数据证据"才真正具有说服力。本项目要回答的核心问题是：

化石数据能否直接"讲故事"，而非仅仅呈现数字？
通过把 Paleobiology Database（PBDB）的数百万条化石记录映射到可交互的时间轴、演化谱系树和古地理地图上，让用户通过交互"触摸"到演化史的形状。
同一场灭绝事件，哪些类群受冲击最大？哪些意外幸存？
"灭绝诊断"面板并排对比事件前后 ±N Ma 内的 8 大门级化石分布，用双向条形图直接显示腕足动物从 890 条跌至 45 条（-95%）这样的具体对比。
灭绝不仅是数量问题，地理分布是否也在收缩？
诊断卡内置按 10° 纬度带切分的化石直方图和前后两张古地理散点图，悬停某一维度会在其他维度上同步高亮——用户能直观发现"赤道带幸存者多、两极几乎清零"之类的规律。
为什么需要 AI 生成图像？
统计数字有说服力，但缺乏"情境"。当用户看到 P-T 事件的双向条形后，立刻可以让 AI 根据 PBDB 的真实数据（代表类群、古纬度、化石数量）生成那个时代的海底生态复原图。从"数据 → 画面"的闭环让"深时间"从概念变成感官体验。
总体目标：做一套"让普通人也能做古生物思考"的交互展台——每次点击、每次拖拽都对应一次真实数据查询，每个可视化元素都在回答一个科学问题。

设计决策与依据

2.1 整体布局 — 四视图联动

采用时间轴（底部）+ 谱系树 · 古地理图 · AI 画廊（顶部三联）的经典 dashboard 布局。 时间作为主导航轴是最关键的决策：所有古生物现象本质上都是时间的函数，把时间放在底部全宽、永久可见，能保证任何交互都有"何时发生"这个锚点。

考虑过的替代方案

方案优点为何未采用
单页 scroll-telling（类似 NYT Graphics） 叙事性强，新手友好 无法支持自由探索；"引导模式"做了简化版本作为补充
3D 地球（Three.js） 视觉冲击大 交互复杂度陡升，性能压力大，且 2D 古地理图已能表达古大陆漂移
Grid 式多视图（R Shiny 风格） 信息密度高 每个视图都小、无法承载精细交互，与"沉浸式"目标相悖

2.2 时间轴 — 分段非线性比例尺

近 5 亿年显生宙（Phanerozoic）的事件密度远高于前 41 亿年的前寒武纪。如果用线性比例尺， 99% 的屏幕会被没有事件的远古占据。采用分段线性比例尺： d3.scaleLinear().domain([4600, 2500, 538.8, 0]).range([0, 18%, 40%, 100%]) 近代展开、远古压缩，视觉上与"地质钟"的感性认识一致。

为什么不用对数比例尺？对数会让年代刻度无法均匀标注（地质纪的边界不是对数分布的）；分段线性可以在每个边界点精确对齐。

2.3 演化谱系树 — 径向 dendrogram

从 LUCA（生命共同祖先）中央向外辐射，几百个分类群节点均匀分布在圆周上。

考虑过的替代方案

  • 水平 dendrogram：节点数超过 50 后横向太窄；径向利用了二维平面
  • Force-directed（力导向）：适合无根网络，但演化树天然有根、层级清晰，力导向反而会掩盖层级关系
  • Icicle / Sunburst：更擅长展示大小 —— 但演化关系的核心是"分支距离"，径向树的径向距离恰好编码了层级深度

灭绝坍缩动画是核心创新：选中灭绝事件后，该时间点之前已灭绝的分支变灰，当时灭绝的分支播放脉冲红光， 存活分支保持绿色。这种"此刻就在发生"的视觉戏剧性，能让用户直观理解灭绝是一个"时间截面"而非抽象概念。

2.4 灭绝诊断卡 — 多维度联动

核心设计：四种维度在同一模态面板中共享 hover 状态。

维度视觉编码联动方式
时间（窗口大小）滑轨改变窗口 → 所有视图重新拉数
分类（门级）横向双向条形hover 某门 → 地图只亮该门的点
空间（古纬度）18 带 ridge 条hover 某带 → 地图只亮该纬度的点
空间（古经纬）迷你散点地图被上述任一维度过滤

考虑过的替代方案

  • Sankey 图（灭绝前门 → 灭绝后门的流动）：对于"前/后静态对比"不够直观；Sankey 强调流动性，而这里需要强调"状态差异"
  • Chord 图：同样偏流动，读者需要学会读图
  • 静态前/后对比卡片（无 hover 联动）：失去"多维度可同时观察"的魅力，退化为信息卡

最终选择双向条形 + ridge + 双地图组合：每种图表都做单一维度的极致表达， 通过共享 state 让不同维度互相解释（hover 一个门的条形 → 地图告诉你"它主要分布在哪里"）。

2.5 状态管理 — Zustand

全局状态（选中的灭绝事件、分类群、时间轴 brush 范围）需要跨组件共享。

  • Redux：样板代码过多，对单人项目杀鸡用牛刀
  • React Context：每次 provider 更新会触发所有订阅者重渲染，无 selector 细粒度控制
  • Zustand（最终）：API 极简、支持 selector 订阅、持久化中间件一行搞定；视图组件只订阅必要字段，避免无关渲染

2.6 动画分工 — D3 vs Framer Motion

D3 负责数学（比例尺、地理投影、brush/zoom 交互、cluster 布局）； Framer Motion 负责视觉过渡（组件进出、数字滚动、卡片形变）。 不让 D3 做 tween 因为 React 会打架；不让 Framer 做 SVG 几何因为表达能力弱。两者清晰分工。

2.7 性能优化权衡

  • SWR 缓存（5 分钟 fresh + 24 小时 stale）：PBDB 响应慢（秒级）而数据变化慢（日/周级），非常适合 stale-while-revalidate。替代方案是无缓存，每次重新拉取——用户体验无法接受
  • rAF 节流：mousemove / d3.zoom 回调每秒触发 ~100 次，直接 setState 会触发全树重渲染；合并到帧边界后 CPU 使用率下降约 60%
  • Web Worker 布局：径向树 `d3.cluster()` 是 O(n²) 计算，对当前规模（~100 节点）虽然只 1-2ms，但将来若支持 PBDB 动态展开（数千节点）就必须异步
  • 图片懒加载 + 渐进模糊反显：AI 图像通常 1024×768+，画廊堆多了会拖慢首屏

3.5 原创与改编声明

本项目的视觉系统、交互逻辑、诊断卡多维联动均为原创设计。径向树的 D3 实现参考了 Observable 社区公开案例 但经过了针对"演化灭绝状态"的大量改造（节点状态五分类、脉冲光环、坍缩动画）。 AI prompt 词库完全自建，包含 30+ 分类群的独有视觉特征描述。

项目按 8 个阶段推进（见 PLAN.md），总计约 55 小时（请根据你的实际情况调整数字）：

阶段内容工时

4.3 最耗时的环节

  1. 灭绝诊断面板的多维度交互同步（~8 h）： 4 个视图、3 个共享 state（hoverPhylum、hoverBand、windowMa）、8 门并发数据请求。 光是"hover 一个门级条 → 两张地图过滤到该门的点 + 纬度带维持不变"的交互逻辑就反复调了 3 次， 包括 state 提升、防抖、z-index 冲突、dim 状态的 CSS cascade 等细节。
  2. PBDB 查询策略调优（~3 h）： 最初对 base_name=Metazoa 的单次大查询持续返回 HTTP 500（PBDB 服务器内部超时）。 尝试过减小 limit、精简 show 字段、加 Vite 代理、内置重试 —— 都未根治。 最终定案"拆成 8 个主要门并发小查询"策略，单门压力可控，部分失败也能出部分结果。
  3. AI prompt 词库构造（~5 h）： 为 30+ 古生物类群写独有的视觉特征描述（形态、纹理、色彩、生态、尺度），并把 PBDB 查到的实际属名、纬度、化石数注入 prompt， 让 AI 生成"带真实数据标注的科研插画"。每个词条都要经过 PBDB 校对和 AI 测试出图。
  4. 径向树坍缩动画 + 五分类节点状态（~4 h）： 节点需要在 alive / dying-now / dead-earlier / dead 四种状态间平滑过渡，且 framer-motion 对 SVG r 属性的初始读取会出现 undefined，需要显式 initial。
  5. 时间轴非线性比例尺 + brush + zoom 共存（~3 h）： 分段线性 + d3.zoom + d3.brushX 三者的坐标系转换容易算错一层，最终采用"所有逻辑坐标都经过同一个 x(ma) 函数"的集中映射风格。

4.4 开发过程的经验与复盘

做对的事：

  • 提前写 PLAN.md：8 个阶段的清晰划分让每次实现都有明确的"完成标准"
  • 坚持 TypeScript 严格模式：诊断面板的跨维度 state 类型完全由编译器保证，重构时几乎零运行时错误
  • D3 与 Framer 分工：避免了"谁来控制 DOM"的经典动画冲突
  • Zustand + selector：几乎不用关心"组件会不会因无关状态而重渲染"

踩过的坑：

  • 最初用 new URL('/api/pbdb/…')（相对路径）在开发环境抛 "Invalid URL"，部署后又出现。一行 base URL 修复，但定位花了 40 分钟
  • Vite 的 dev proxy 只在开发时有效，生产 Pages 站点无法用同样机制调 AI 服务 —— 这是本项目生产部署的已知局限（需配额外的 edge proxy）
  • Framer Motion 在 SVG <circle> 上的 r 属性动画需要显式 initial，否则会从 DOM 读到 undefined 导致抖动
  • TypeScript 的 DedicatedWorkerGlobalScope 需要在 Worker 文件顶部加 /// <reference lib="webworker" />，否则 CI 环境下 tsc -b 会报错（本地 IDE 偶尔静默通过）

4.5 下一步可能的改进

  • 为 AI 绘图部署一个 Cloudflare Workers 代理，让生产站点也能生成图
  • 增加植物界灭绝诊断（目前诊断只看动物界 8 大门）
  • 接入 Macrostrat API 获取真实古大陆轮廓，替换当前的经纬散点

  • 移动端专门的单列布局（目前 860px 以下响应式已基本可用但密度偏高）
